# 阅读节律

这篇不是工程规格，而是说明 Hermes 为什么要这么小心。书籍自动化的目的不是制造更多按钮，而是让阅读时不用想格式、传输和更新风险。

## 一天里的位置

白天处理书，晚上读书。Hermes 负责把 TXT/EPUB 整理成可读版本，WebDAV 把它放到 KOReader 能看到的地方。真正坐下来读的时候，系统应该安静。

典型流程：

```text
白天：下载或整理源文件
  -> Hermes intake
  -> 查看 reports
  -> published 或 pending

晚上：KOReader 打开 /books 里的 EPUB
  -> 阅读、标注、书签
  -> 不在阅读时思考格式转换
```

### 在线、离线与静默

手机把书发给 Hermes 时，云端接收不等于本机处理。Windows 离线则回复 `received` 或 `queued`，不生成假的质量报告；重新上线后先检查源文件 hash、目标书是否变化以及任务是否仍需要执行，再交给本地管线。网络或 WebDAV 不可用时保留本地 runs 和报告，不用无限重试打扰阅读。

进入晚间阅读后，书务任务默认让路：不在正在读书时覆盖旧版、不连续推送元数据建议，也不为了清空队列启动大模型或正文分析。新书处理可以等到下一次维护窗口；正在阅读的旧书永远以进度和标注保护为先。

状态交接只说事实：`published` 是发布器已完成，`pending` 是候选受保护，`blocked` 是门禁未过；重新读取远端文件后才能标记为 `verified`，而 Kindle/KOReader 已经下载仍是设备端事实，不能由云端推断。

## 为什么旧书更新要谨慎

阅读进度和标注不是抽象地“跟着书走”。它们依赖路径、文件名、书籍状态文件、内部锚点，有些配置还依赖内容 hash。

所以 Hermes 宁愿多 pending，也不应该静默覆盖旧书。一本正在读的书，如果章节 href、item id、图片资源或 CSS 依赖变了，标注位置可能漂移。对阅读体验来说，这比多点一次确认更糟。

## 当前自动化边界

已经自动：

- 本地 TXT/EPUB intake。
- EPUB 检查和报告。
- 元数据增强框架。
- 安全发布或 pending。
- KOReader `hashdocsettings` 风险阻断。

仍需人工或后续实现：

- 从站点抓取原始书籍。
- 判断某本书是否缺章。
- 删除正文广告。
- 补插图。
- 处理 KOReader 本地 sidecar。
- 把标注稳定地合成为知识库笔记。

这个边界很重要。现在的 Hermes 先保证“不会把你的阅读状态搞坏”，再逐步增加智能。

## 使用节奏

### 新书

新书最轻松。没有旧进度，`NEW_BOOK` 发布即可。如果缺封面或 metadata 不完整，报告会写出来，但不会影响你先读。

### 连载更新

连载更新看 `update-diff.md`：

- `SAFE_APPEND`：旧章节稳定，新章节追加，适合直接更新。
- `BLOCKED_RISKY`：旧章节有变化，先看原因。

网文站点常见“前文修订”“广告插入”“章节标题变动”。这些对人眼不一定明显，对标注锚点却可能很危险。

### 元数据修订

书名、作者、插画师、系列、ISBN、封面这些可以自动补，但不该改变 WebDAV 文件名和主 identifier。Hermes 会把外部搜索与模型判断留在 `metadata-report` 里，方便回头追溯。

## 标注回流

本仓库已有 KOReader 标注解析脚本，但完整的“每日自动拉取、去重、总结、写入知识库”还不应写成已落地事实。当前可靠说法是：

- KOReader 可以产生标注。
- 标注可以通过插件或文件同步导出。
- 后续 Hermes 可以读取这些导出文件，进入笔记处理管线。

这部分会单独设计，不和 book intake 混在一起。

## 设计取舍

Hermes 的取舍很朴素：

- 自动化优先处理重复劳动。
- 旧书覆盖永远保守。
- 大模型用于判断，不直接接管正文。
- 每次运行都留下报告。

如果某一步需要你停下来确认，那通常不是系统不够智能，而是它在保护已经读过的书。
